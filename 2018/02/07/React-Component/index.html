组件生命周期

每个组件都有几个生命周期方法，您可以在进程中的某些特定时间重写以运行代码。以'will'为前缀的方法在事件发生之前被调用，以'did'为前缀的方法在事件发生之后被调用。

Mounting 装载

当创建一个组件的实例并将其插入到DOM中时，将调用这些方法：

  • constructor()
  • componentWillMount()
  • render()
  • componentDidMount()

Updating 更新

更新可以由props或state态的更改引起。 当一个组件被重新渲染时，这些方法被调用：

  • componentWillReceiveProps()
  • shouldComponentUpdate()
  • componentWillUpdate()
  • render()
  • componentDidUpdate()

Unmounting 卸载

从DOM中删除组件时调用此方法：

  • componentWillUnmount()

Error Handling 错误处理

在渲染期间，生命周期方法或任何子组件的构造函数中发生错误时调用此方法。

  • componentDidCatch()

Other APIs

每个组件还提供一些其他API：

  • setState()
  • forceUpdate()

Class Properties 类的属性

  • defaultProps
  • displayName

Instance Properties 接口属性

  • props
  • state

Reference

render()

render()方法是必需的。

调用时，应检查this.props和this.state并返回以下类型之一：

  • React elements 一般通过JSX创建。标签可以是原生的DOM组件(<div />)，也可以是自定义的复合组件(<MyComponent />)。
  • String and numbers 这些在DOM中被渲染为文本节点。
  • Portals 由ReactDOM.createPortal创建。
  • null 什么都不渲染，或者渲染为空。
  • Booleans 什么都不渲染，或者渲染为空。(大多存在支持return test && <Child />模式，其中test是布尔值)

当返回值为null或false，ReactDOM.findDOMNode(this)将返回null.

render()函数应该保持纯净(render()是个纯函数)，意味着不会修改组件的状态，每次调用它都返回相同的结果，并且不会直接与浏览器交互。如果你需要与浏览器直接交互，请在componentDidMount()或者其它生命周期的方法中执行你的工作。保持render()函数的纯洁性使组件更容易理解。

注意

如果shouldComponentUpdate()返回false，render()将不被调用。

constructor()构造函数

React组件的构造函数在挂载之前被调用。 在实现React.Component子类的构造函数时，应该在任何其他语句之前调用super（props）。 否则，this.props在构造函数中将是undefined，这会引起bugs。

在构造函数内避免引入任何副作用和订阅(subscriptions)。对于这些用例，使用componentDidMount()代替。(这一段的翻译狗屁不通)

构造函数是初始化state的正确位置。因此，只需给this.state分配个对象；不要尝试从构造函数里调用setState()。构造函数也经常用于为类实例绑定事件处理程序(event handlers)。

如果你不初始化state也不绑定方法，你不需要为你的React组件实现一个构造器。

小心这种模式，因为任何props更新都不会使state为最新状态。而不是将props同步到state，你经常想要解除状态提升(lift the state up)。

如果你想fork props来给state使用，你可能要实现componentWillReceiveProps(nextProps)来保持state为最新状态。但是提升状态通常更容易，并且更少出现错误。

componentWillMount()

在组件安装发生(mounting occurs)之前立即调用componentWillMount()。它是在render()方法之前调用。因此在此方法中同步调用setState（）不会触发额外的渲染。一般来说，我们建议使用constructor()替代。

自己的理解

componentWillMount()是在render()之前调用，并且在componentWillMount()中使用setState()是不会触发界面重新渲染的。官方建议使用constructor()方法替代componentWillMount()的一些作用。

在这个函数内避免引入任何副作用和订阅(subscriptions)。对于这些用例，使用componentDidMount()代替。

这是在服务端渲染上调用的唯一的生命周期钩子。

componentDidMount()

在组件安装(mounted)后立即调用componentDidMount()。需要DOM节点的初始化都应在这里。如果你需要从远程端点加载数据，这里是一个初始化网络请求的好地方。

这个方法也是一个订阅(subscriptions)的好地方。如果你在发出了订阅，别忘了在componentWillUnmount()里取消订阅(unsubscribe)。

在这个方法里调用setState()将触发额外的渲染。但这个方法会在浏览器更新屏幕之前发生。这保证即使render()在这种情况下被调用两次，用户也不会看到中间状态。要小心的使用这种模式因为这常常是引起性能问题的原因。It can, however, be necessary for cases like modals and tooltips when you need to measure a DOM node before rendering something that depends on its size or position.

componentWillReceiveProps()

componentWillReceiveProps()在组件装载后(mounted)接收新props前被调用。++如果您需要更新状态以响应prop更改++(比如，重置状态)，你可以比较this.props()和nextProps()，并在此方法中使用this.setState()执行状态转换。

请注意，即使props尚未更改，React也可能调用此方法，因此如果您只想处理更改，请务必比较当前值和下一个值。父组件导致组件重新渲染时，可能会发生这种情况。

在装载时或装载中(during mounting)，React不会调用componentWillReceiveProps()和初始化props。只有在一些组件的props要更新时才会调用此方法。调用this.setState()一般不会触发componentWillReceiveProps()。

shouldComponentUpdate()

使用shouldComponentUpdate（）让React知道组件的输出是否不受当前state或props变化的影响。 默认行为是在每次状态更改时重新呈现，在绝大多数情况下，您应该依靠默认行为。

在组件渲染前，接收到新的props或state时会调用shouldComponentUpdate()。返回值默认是true。初始化render()或当forceUpdate()被应用时不会调用此方法。
